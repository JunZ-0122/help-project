# 聊天链路手工联调清单

更新时间：2026-03-12

## 1. 联调范围

当前聊天链路已接通以下接口：

- `GET /api/chat/{requestId}`：读取聊天历史
- `POST /api/chat/`：发送文本消息
- `PUT /api/chat/read`：标记已读

当前仍未接入：

- WebSocket 实时收发
- 实时未读数 / 会话自动刷新

## 2. 联调前提

- 后端 `help` 已启动，接口基址为 `http://localhost:8080/api`
- 前端 `help_system` 已安装到预览器或真机
- 求助者账号、志愿者账号都可以正常登录
- 已存在一条“已分配给志愿者”的求助单
- 该求助单同时满足：
  - 求助详情页可以看到志愿者信息
  - 志愿者端可以在“我的订单 / 进度页”打开对应服务单

建议使用同一条 `requestId` 全程验证。

## 3. 入口检查

### 求助者入口

路径：

- `pages/help-seeker/RequestDetailPage`

检查项：

- 已分配状态下出现“发送消息”按钮
- 点击后跳转到 `pages/volunteer/ChatPage`
- 路由参数里带有 `requestId`
- 若详情页已拿到志愿者信息，还应带有 `volunteerId`、`volunteerName`

### 志愿者入口

路径：

- `pages/volunteer/ProgressPage`

检查项：

- 点击“发送消息”后跳转到 `pages/volunteer/ChatPage`
- 路由参数里至少带有 `id`，并能被聊天页识别为 `requestId`

## 4. 聊天页加载检查

页面：

- `help_system/entry/src/main/ets/pages/volunteer/ChatPage.ets`

首次进入时应验证：

- 页面能从本地存储读到当前登录用户 `user`
- 页面会额外请求一次求助详情，用于确定聊天对象
- 求助者进入时，聊天对象应解析为对应志愿者
- 志愿者进入时，聊天对象应解析为对应求助者
- 页面标题显示聊天对象姓名，而不是默认“聊天”
- 历史消息能够正常显示，时间格式为 `HH:mm`

异常检查：

- 如果本地没有 `user`，页面应提示登录状态异常
- 如果 `requestId` 缺失，页面不应崩溃
- 如果求助单没有分配志愿者，求助者侧发送按钮应不可用

## 5. 历史消息与已读验证

### 步骤 A

- 用求助者发送 1 条消息
- 退出聊天页
- 切换到志愿者账号进入同一 `requestId` 的聊天页

### 预期

- 志愿者能看到求助者刚发送的消息
- 进入页面时会调用 `PUT /api/chat/read`
- 仅“接收方是当前用户且未读”的消息会被标记

### 步骤 B

- 志愿者退出聊天页，再次进入同一聊天页

### 预期

- 已读消息不会重复触发异常
- 历史列表顺序稳定

## 6. 双向发送验证

### 求助者 -> 志愿者

- 求助者进入聊天页
- 输入文本并发送

预期：

- 消息发送成功后立刻追加到当前页面列表
- 输入框清空
- 无重复消息

### 志愿者 -> 求助者

- 志愿者进入同一 `requestId` 聊天页
- 输入文本并发送

预期：

- 消息发送成功后立刻追加到当前页面列表
- 返回求助者侧重新进入后可看到该消息

## 7. 关键回归点

- 求助者详情页里的“发送消息”仍可正常打开聊天页
- 志愿者进度页里的“发送消息”仍可正常打开聊天页
- 聊天页标题在两种角色下都能解析正确
- 求助者不会把消息发给自己
- 志愿者不会把消息发给自己
- 空消息不能发送
- 无 `targetUserId` 时发送按钮应保持禁用

## 8. 当前已知限制

- 发送后另一端不会实时收到，需要重新进入页面或手动重开链路验证
- 未读数没有独立 UI 展示，只能通过接口和已读行为间接确认
- 聊天对象解析依赖求助详情接口返回的 `userId` / `volunteerId` / `volunteerName`

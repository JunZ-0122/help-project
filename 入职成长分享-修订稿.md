# 入职成长分享

## 写在前面：接下来三个月，我想往哪走

先简单抛一下我对**接下来三个月**的打算，后面再聊这三个月里发生了什么。

我希望自己能从「单点执行」往「能扛一块事」靠：不仅跟着把开发做完，而是尽量从需求理解到上线走完整，卡住了也主动推进；同时，也希望在接下来的三个月里，**能稳定负责至少 1 个模块的完整交付，并对结果负责，而不是只参与过程。**动手前先把链路、上下游和边界拆清楚，减少反复试错；在把问题讲清楚的前提下，用 AI 放大效率，结果不理想时先从自己的输入和调整上找原因。

公司讲 10x、讲提速，我也会**把个人目标设高一点**（比如往设计、前端、后端、测试、部署都能沾一点的方向看），哪怕三个月里只往前走了 20%、30%，我觉得也是实打实的进步；能先把当前模块里能承担的前后端协作吃透，也是很好的台阶。

今天更像**分享**：我业务线上的细节，在座不同岗位的同事不一定都熟悉，我会尽量把**结论和感受**讲清楚；如果中间有哪段让你想起自己刚上手时的状态，或者能给同届实习的同学一点共鸣，我会特别开心。

## 一、分享背景

刚进入项目时，我面对的是一个业务复杂的系统。和在学校写代码不同，在真实项目中，代码不是写完就结束了，它需要长期运行、持续变化，也需要被很多人一起维护。

所以这三个月对我来说，最大的变化并不是学会了多少技术，而是开始慢慢改变自己看问题的方式。

这三个月里，我主要参与了 3 个模块：

- 奇门回调
- JDL 推送与回调
- 销售出库

回过头看，这三个月更像是一个从「先把功能做出来」→「开始理解系统、理解业务、理解协同」的过程。

## 二、第一个月：奇门回调模块

### 从「直接写代码」到「开始理解结构设计」

刚进入公司的第一个月，为了让我熟悉系统和业务，先让我通过阅读代码来理解整个模块的结构。

当时给到我的模块是奇门对接相关的代码。奇门对接主要包括推送和回调两部分，其中推送模块已经完成了一次重构，而回调模块还没有整理过。我尝试对回调模块进行一次结构整理和重构。

在阅读代码之后，我发现奇门的推送和回调在整体流程上其实比较相似，于是我产生了一个比较「理想化」的想法：能不能设计一套通用的回调处理框架，让不同类型的回调都能复用同一套结构。

当时也给了我一个建议：这个模块本身不复杂，关键是先把结构搭好，后面的扩展就会比较顺。

于是我就带着「做一个通用框架」的想法，开始直接写代码。

但在第一次代码 Review 的时候，我很快遇到了问题。虽然代码可以跑通，但整体结构并不好：想做通用，但真正可复用的部分并没有抽象出来；不同职责的逻辑混在一起，代码显得比较臃肿；一些处理逻辑写死在代码里，扩展性也不好。也就是说，当时我虽然是想做「通用设计」，但实际上只是把很多逻辑堆在了一起，并没有真正把结构拆清楚。

这件事也让我多想了想**平时怎么接别人的 Idea**：先接纳、再自己想一遍合不合理、有没有更好的方案，如果有更好的想法就主动拿出来和团队碰，而不是觉得「别人说的对」就按任务执行到底——后者很容易变成只堆实现、少了自己的判断。

后来在  的帮助下，我们重新梳理了一遍整个回调处理流程，包括：回调处理流程应该如何拆分；哪些逻辑属于通用部分；哪些逻辑应该交给具体业务实现；不同单据类型的回调应该如何扩展。

在重新设计结构之后，我对代码进行了第二次调整。虽然这个过程前前后后花了一些时间，但结构确定下来之后，后面在开发其他单据类型回调的时候，整体开发速度明显提升了很多。

在奇门回调模块中，我不仅完成了功能开发，还做了一些技术层面的探索与总结：

- **设计层面**：尝试将回调流程抽象为通用框架，最终落地为「模板方法 + 策略模式 + 注册中心」组合设计，提升扩展性
- **稳定性层面**：针对回调重复推送问题，引入分布式锁机制，保证幂等性
- **对接层面**：总结了一套「文档 → 映射 → 验证」的方法论，并在后续 JDL 对接中复用
- **沉淀层面**：输出设计文档，将经验结构化，而不是停留在代码层

这段经历让我第一次比较深刻地意识到：「写代码」和「设计结构」其实是两件完全不同的事情。一开始我只是想着把功能实现出来，但在这个过程中我开始慢慢理解：如果结构没有想清楚，代码很容易越写越复杂；但如果职责拆分合理，后续扩展反而会变得更简单。

所以第一个月对我来说，最大的收获并不是完成了这个模块的开发，而是让我第一次意识到：在真实项目中，很多时候代码不是写出来的，而是设计出来的。这也让我开始慢慢从直接写实现 → 先思考结构再写代码。

## 三、第二个月：JDL 推送与回调

### 从「想复用代码」到「理解差异与异步流程」

第二个月，我主要参与的是 JDL 推送与回调模块。

一开始，我有一个很自然的想法：既然同样是 WMS 对接能力，整体逻辑应该类似，可以复用之前奇门的设计。但真正接触后我很快发现，这种「看起来差不多」的判断，其实只停留在表面。

JDL 和奇门在几个关键点上存在明显差异：

- **协议不同**：奇门基于 XML + HTTP，JDL 基于 JSON + SDK
- **时序不同**：奇门是同步回调，JDL 通过 MQ 异步回调
- **架构不同**：JDL 引入 MessageOutbox + MQ + Temporal 工作流
- **机制不同**：对失败、重试、补偿的要求更复杂

这让我第一次比较直观地意识到：**系统对接的难点，不在「接口调通」，而在「处理差异」。**

刚开始其实是有点被这些概念“砸到”的，但慢慢拆开之后，会发现它们其实都是在解决一些很具体的问题。

在具体实践中，我主要做了三件事：

1. **对接 JDL API 文档，形成方法论**
  这是我第二次对接第三方 WMS，但协议完全不同。我没有再「边看文档边写代码」，而是整理出一套流程：文档 → 字段映射 → 样例验证，并把关键字段与系统内部模型建立对应关系。这让我意识到：对接能力不在于记住接口，而在于形成可复用的方法。
2. **理解异步架构和链路设计**
  JDL 的处理链路是：Controller → 写入 Outbox → 发 MQ → 异步消费 → 更新状态；推送流程则通过 Temporal 工作流执行，Controller 只负责触发。在这个过程中，我开始真正理解异步设计要解决的问题：主流程与异步流程如何拆分、失败后如何补偿、重复消息如何保证幂等、状态何时才算最终完成。也逐渐理解：很多「复杂设计」，本质是在对抗不确定性。
3. **理解「抽象边界」而不是强行复用**
  在对比奇门和 JDL 的实现后，我发现：共性可以抽象（如回调处理流程），差异必须隔离（协议、时序、架构）。如果强行复用，反而会让系统更复杂。我开始不再只关注「代码能不能复用」，而是开始思考：场景之间的本质差异是什么；设计是否能应对异常和变化；系统在复杂情况下是否仍然稳定。设计的意义，不是让代码看起来统一，而是让系统在变化和异常中依然稳定。

在这个模块中，我也做了一些相对系统性的技术总结，而不是只停留在功能实现：

- 梳理并理解了基于 Outbox + MQ 的异步架构，明确了「先落库再消费」在高并发和重复消息场景下的意义
- 理解了 Temporal 工作流在推送链路中的作用，以及主流程与异步流程的职责拆分
- 在实践中总结了异步系统需要重点关注的几个点：幂等、重试、补偿、最终一致性
- 对比奇门与 JDL 的实现差异，形成了「共性抽象 + 差异隔离」的设计认知，而不是强行复用

也让我在之后面对类似系统时，会更优先去分析差异本身，而不是一开始就想着复用已有方案。

## 四、一个小转折：从「排斥 AI」到「开始拥抱 AI」

在第二个月到第三个月之间，其实还有一个对我影响比较大的变化，就是我开始重新看待 AI 在开发中的作用。

刚进入公司的时候，作为一个没有太多项目经验的实习生，我其实对 AI 是有些排斥的。当时我的想法很简单：既然刚开始做开发，我更希望通过自己写代码来积累经验，而不是过早依赖工具。所以一开始即使团队在讨论 AI 提升开发效率，我也更多是保持一种观望的态度。

在每周的 xpa 例会上， 其实多次和大家分享过如何利用 AI 来辅助开发，也经常提到一句话：「我现在基本上一行代码都不写。」刚开始听到这句话的时候，我其实很难真正理解这句话背后的含义。

直到后来在公司的 2026 年度导向会议之后提出的 10x 理念，我开始重新思考这个问题。我逐渐意识到，AI 的意义并不是替代开发者，而是改变开发者的工作方式。

以前我的开发方式更多是：自己写代码、自己调试、自己一点一点解决问题。

这段时间的一个明显变化，是我开始把 AI 当成「协作的一部分」，而不是单独的工具。在实际开发中，我逐渐形成了一种更清晰的使用方式：**先自己把业务场景、链路和边界想清楚，再借助 AI 去做分析、拆解和落地。**

比如在一次前后端联调问题中，我先用 GPT 梳理完整业务场景并生成结构化文档，再交给 Claude Code 去做代码定位和修复，最后再生成对接文档，整个过程明显更顺，也减少了重复沟通成本。

**相关文档**：出库单修改物流 — 前端联调文档

这个过程让我意识到：AI 并不会替代思考，反而要求我把问题表达得更清楚、拆得更细。我也开始从「自己写代码」，慢慢转向「设计 + 判断 + 协作」，让 AI 去放大效率，而不是替我做决定。

如果说以前我更习惯于通过写代码来完成任务，那么现在我开始尝试通过设计、审阅和判断来推动开发。某种程度上来说，我也在慢慢理解：接纳，是从抵触到理解，从重复操作到思考价值的自我升级。

这也让我开始思考，在 AI 时代，开发者需要具备的能力，可能不仅仅是编码能力，而是系统理解能力和问题判断能力。

## 五、第三个月：销售出库模块

### 从「追着问题跑」到「理解业务与协同」

第三个月，我主要参与的是销售出库模块。这个模块比前两个月更贴近核心业务，它连接了订单、库存、仓库，以及出库完成后的状态流转。

刚开始接手时，我最大的感受是：问题很多，而且总是不断冒出来。

当时我的工作方式基本上是：发现问题 → 定位代码 → 修改逻辑 → 验证。很快我意识到，自己总是在追着业务后面跑。

表面上看，问题出现在销售出库模块，但真正的原因往往不止于这一段代码，可能涉及订单、库存、仓库配置，甚至 WMS 回调。为了理解问题，我开始更多地与  沟通，一起梳理完整业务流程。

这个过程让我慢慢意识到：

- 销售出库不是孤立功能，而是整条业务链路中的一个环节
- 许多问题不是单点问题，而是链路问题
- 只盯着报错点改代码，很容易一直落在业务后面
- 真正的解决方案，需要人与人之间的协作，而不仅仅是改代码

与此同时，我也发现了一个问题：如果只是在嘴上的沟通，很难落地到问题的切入点上。我也在思考一种可能的优化方式：通过文档明确模块输入、输出、状态流转和异常处理，让团队在对接和讨论时有统一参考，也方便后续复盘和协作。

这也让我在之后处理问题时，会更习惯先看整条业务链路，而不是只盯着当前模块。

如果说前两个月让我开始理解代码结构和系统差异，那么第三个月让我真正体会到：**写业务代码，不只是实现逻辑，更是理解链路、理解团队、理解如何高效协作。**

## 六、全员协同课

入职三个多月，刚好参与了全员协同课，对我来说更像一次及时的思维校正。

在此之前，我还带着比较明显的学生惯性：更多是按照任务执行，完成后交付，很少去思考这件事情本身要解决的核心问题是什么。

课程中提到「成果导向不是为了完成，而是为了超越原有条件」，让我开始重新审视自己的做事方式。比如在整理会议纪要时，不再只是记录内容，而是会有意识地去提炼关键信息、明确后续动作，帮助团队更快推进事情落地。

在协同方面，我也有了新的认识。之前更习惯于把自己的事情做好，但在实际工作中逐渐意识到，很多结果是通过多人协作完成的，节奏对齐和信息同步往往比单点优化更重要。

同时，在面对问题或做得不理想的情况时，我的处理方式也发生了一些变化。相比之前容易陷入原有路径，现在会更倾向先判断方向是否正确，再决定是否调整方案，而不是一味在细节上反复尝试。

虽然这门课没有直接教授具体技术能力，但它让我从「被动完成任务」逐渐转向「主动推动结果」，对我从学生向工程师转变起到了比较关键的作用。

## 七、未来三个月的成长导向

接下来三个月，我希望自己完成一个比较明确的转变：从「单点执行」到「模块 Owner」的全链路蜕变。

**第一，是敢承担，而不是只参与。** 不再只是跟着做开发，而是主动去接一块相对完整的模块，从需求理解到上线尽量走完整。在遇到不确定或卡点时，也能主动推进，而不是等安排。目标是逐步做到：不仅把事情做完，也能对结果负责。

**第二，是先把问题拆清楚，再动手。** 之前有时候会直接上手改代码，后面发现效率不高。接下来会更有意识地先把链路、上下游和边界梳理清楚，把问题拆细之后再推进。尽量减少反复试错，让问题定位更有方法，而不是靠感觉。

**第三，是提升与 AI 的协作方式。** 在把问题拆清楚的基础上，通过更清晰的输入，让 AI 帮助做分析、定位和整理。如果结果不理想，也会先从自己的输入和理解去调整，而不是简单认为工具不行。慢慢形成一套稳定的方式：我负责把问题讲清楚，AI 帮我放大效率。

**第四，是自驱再往外露一点、向内观再多一点。** 比如会上有意识地多表达自己的想法；对团队里相对新的模块多保持一点探索欲，不把自己限在「只认识自己那一块」；定期想想自己还有哪些习惯需要调整，避免只在舒适区里打转。

也希望自己在一些没有明确分配的事情上，能更主动站出来，而不是等任务落到自己头上。

## 八、总结与感谢

回顾这三个月，我的技术变化大致经历了三个阶段：

- **第一个月**：以「实现功能」为导向，开始接触结构设计，但对抽象和职责划分理解不够深入
- **第二个月**：开始理解系统差异与架构设计，能够区分「可复用」和「必须隔离」的边界
- **第三个月**：逐步从代码实现转向业务链路理解，开始从「改代码」转向「定位系统问题」

从结果来看，我的变化主要体现在：从「写代码」→「设计结构」；从「关注实现」→「关注系统稳定性」；从「单点问题」→「全链路思考」。这也意味着，我开始从一个执行开发，向具备工程思维的开发者转变。

当然，这段成长不仅仅来自于工作本身，也来自于团队的氛围。

加入 @XPA 项目组之后，我有一个很直观的感受：虽然大家在会议上有时候会比较安静，但在日常工作交流中，每个人其实都有很多自己的想法，也经常会分享一些有意思的事情。很多问题在沟通和讨论中慢慢被拆开，也让协作变得越来越顺畅。  和  组织团建活动，让大家在紧凑的工作节奏之外有机会放松下来，也慢慢打破了团队那种紧绷的状态。

除了日常工作交流之外，团队里其实也有很多轻松有趣的小群，比如：

- **协同干饭群**：  经常会在午饭前提醒大家不要忘记每日导向，  也会紧接着发出「外卖」两个字，提醒大家该点饭了。
- **教主群**：  会在群里分享可以「薅羊毛」的 AI 工具，大家也会聊一些关于 AI 的新东西或者有趣的事情。有时候一个小工具或者一个新玩法，很快就会在群里被大家尝试一圈。
- **健身群**：  会带着大家一起健身，在挥汗的同时，既减去了体重，也卸下了一天工作的疲惫。

这些看起来只是工作之外的一些小事情，但其实也在慢慢建立一种更自然的协同方式。当大家彼此更熟悉、更放松的时候，在工作中沟通问题、讨论方案、一起解决复杂问题也会更加顺畅。

所以对我来说，这三个月不仅是在学习如何做开发，也是在慢慢适应一个团队、融入一个团队，并在这样的环境中不断成长。如果今天的分享里，有哪一点让你觉得「我也经历过」或者「我也想试试」，我会特别高兴——也欢迎同届的同学会后一起聊。

谢谢大家！

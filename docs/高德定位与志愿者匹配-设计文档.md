# 高德 API 接入与志愿者匹配 · 设计文档

## 1. 需求概述

- **求助者定位**：接入真实高德 API，实现用户定位；求助者发布求助时上报/选择位置，便于展示与匹配。
- **志愿者匹配**：后端根据求助位置，通过算法筛选“附近”的志愿者（如按距离、接单量、评分等），支持社区分配与志愿者端“附近求助”列表。
- **数据库扩展**：在现有 5 张核心表基础上，增加与定位、匹配、运营、审计相关的表与字段，支撑后续功能。

---

## 2. 高德 API 接入设计

### 2.1 高德产品选型


| 场景              | 推荐产品                                  | 说明                         |
| --------------- | ------------------------------------- | -------------------------- |
| 前端定位（HarmonyOS） | 高德 定位 SDK（若支持鸿蒙）或 **JS API / Web 服务** | 鸿蒙端可用系统定位 + 高德逆地理/地理编码补齐地址 |
| 地址 ↔ 坐标         | 高德 **Web 服务**：地理编码 / 逆地理编码            | 后端或前端把“地址”转经纬度，或把经纬度转“地址”  |
| 距离/路线           | 高德 **Web 服务**：距离测量、路径规划（可选）           | 后端计算求助点与志愿者距离，或预估到达时间      |


**密钥**：在高德开放平台创建应用，获取 **Web 服务 Key**（后端调用）、可选 **JS Key**（若前端用 H5 高德地图）。Key 仅配置在后端与构建配置中，不提交到代码库。

### 2.2 前端（help_system）

- **定位能力**  
  - 使用 HarmonyOS **@ohos.geolocation** 获取设备经纬度（用户授权）。  
  - 可选：将 (lat, lng) 发往后端，由后端调高德逆地理得到省市区 + 详细地址，再回显给用户确认/编辑。
- **发布求助**  
  - 提交时带 `latitude`、`longitude`、`location`（文字地址）。  
  - 若用户只填地址不选点，可由后端用高德地理编码补全经纬度（见下）。
- **展示**  
  - 列表/详情可展示“距离我 X km”（需当前用户位置，或后端返回与某点的距离）。  
  - 后续若做地图选点，可引入高德地图组件（按鸿蒙生态可用性选型）。

### 2.3 后端（help）

- **配置**  
  - `application.yml` 中增加：`amap.web-service-key`（从环境变量或配置中心读取，不写死）。
- **服务封装**  
  - `AmapService`：  
    - 地理编码：地址 → (lat, lng)；  
    - 逆地理：(lat, lng) → 省市区 + 详细地址；  
    - 距离测量：两点直线距离或调用高德距离测量 API（按需）。
  - 对高德 HTTP 接口做封装，统一异常与重试（可选）。
- **与业务结合**  
  - 创建/更新求助时：若仅有地址无经纬度，则调地理编码补全 `latitude`、`longitude`。  
  - 志愿者匹配/附近求助：基于 `help_requests.latitude/longitude` 与志愿者位置（见下“用户位置”）算距离，排序后返回。

### 2.4 志愿者匹配算法（概要）

- **输入**：求助请求 `request_id`（或经纬度）、可选筛选（类型、状态等）。  
- **志愿者位置**：来自 `user_locations` 表（最近一次上报的经纬度 + 更新时间），或从志愿者历史接单地址推算（见数据库扩展）。  
- **步骤**：  
  1. 取求助点 `(lat, lng)`。
  2. 筛选“可接单”的志愿者（状态、角色、当前订单数等）。
  3. 计算每个志愿者与求助点的距离（Haversine 或高德距离 API）。
  4. 按距离排序，可选：叠加评分、完成单量等权重。
  5. 返回 Top N 或分页列表，供社区分配或志愿者端“附近求助”使用。

## 3. 数据库扩展设计

### 3.1 现有表与可扩展点

- **users**：已有 `address` 等；可增加 `latitude`、`longitude`、`location_updated_at`（或改用 `user_locations` 存实时位置，见下）。  
- **help_requests**：已有 `latitude`、`longitude`、`location`，满足求助点存储。  
- **volunteer_orders**：已有 `latitude`、`longitude`，可保留；若需更细的“服务地点”，可后续加字段或关联地址表。

### 3.2 新增表建议


| 表名                         | 用途                              |
| -------------------------- | ------------------------------- |
| **user_locations**         | 用户最近位置（志愿者/求助者上报），用于距离计算与“附近”展示 |
| **volunteer_availability** | 志愿者可服务时间段（可选），用于“可接单”筛选         |
| **service_areas**          | 志愿者服务范围（行政区或半径），用于粗筛            |
| **notifications**          | 站内通知（系统推给用户的消息列表）               |
| **audit_log**              | 关键操作审计（谁在何时做了什么）                |
| **system_config**          | 系统配置键值（如开关、限流参数、高德 Key 名等）      |
| **request_views**          | 求助被查看/点击统计（可选，用于热度或推荐）          |


以下给出各表字段设计与建表 SQL 草案，可按需删减或分阶段建表。

## 4. 新增表结构（SQL）

以下 SQL 为草案，需在测试环境执行并核对后再上生产。

```sql
-- -------------------------------
-- 用户位置（最近一次上报）
-- -------------------------------
CREATE TABLE IF NOT EXISTS user_locations (
  id           VARCHAR(64) PRIMARY KEY,
  user_id      VARCHAR(64) NOT NULL,
  latitude     DECIMAL(10, 7) NOT NULL,
  longitude    DECIMAL(10, 7) NOT NULL,
  address      VARCHAR(512),
  source       VARCHAR(32),
  created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uk_user (user_id)
);
CREATE INDEX idx_user_locations_user_id ON user_locations(user_id);

-- -------------------------------
-- 志愿者可服务时间（可选）
-- -------------------------------
CREATE TABLE IF NOT EXISTS volunteer_availability (
  id           VARCHAR(64) PRIMARY KEY,
  user_id      VARCHAR(64) NOT NULL,
  day_of_week  TINYINT NOT NULL COMMENT '1-7',
  start_time   TIME NOT NULL,
  end_time     TIME NOT NULL,
  created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at   DATETIME ON UPDATE CURRENT_TIMESTAMP
);
CREATE INDEX idx_vol_avail_user ON volunteer_availability(user_id);

-- -------------------------------
-- 志愿者服务范围（行政区或半径）
-- -------------------------------
CREATE TABLE IF NOT EXISTS service_areas (
  id           VARCHAR(64) PRIMARY KEY,
  user_id      VARCHAR(64) NOT NULL,
  type         VARCHAR(16) NOT NULL COMMENT 'radius|district',
  center_lat   DECIMAL(10, 7),
  center_lng   DECIMAL(10, 7),
  radius_km    DECIMAL(6, 2),
  district_code VARCHAR(32),
  district_name VARCHAR(128),
  created_at   DATETIME DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_service_areas_user ON service_areas(user_id);

-- -------------------------------
-- 站内通知
-- -------------------------------
CREATE TABLE IF NOT EXISTS notifications (
  id           VARCHAR(64) PRIMARY KEY,
  user_id      VARCHAR(64) NOT NULL,
  title        VARCHAR(128) NOT NULL,
  body         TEXT,
  type         VARCHAR(32),
  ref_id       VARCHAR(64),
  ref_type     VARCHAR(32),
  is_read      TINYINT(1) DEFAULT 0,
  created_at   DATETIME DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_notifications_user_read ON notifications(user_id, is_read);
CREATE INDEX idx_notifications_created ON notifications(created_at);

-- -------------------------------
-- 操作审计日志
-- -------------------------------
CREATE TABLE IF NOT EXISTS audit_log (
  id           BIGINT AUTO_INCREMENT PRIMARY KEY,
  user_id      VARCHAR(64),
  action       VARCHAR(64) NOT NULL,
  resource_type VARCHAR(32),
  resource_id  VARCHAR(64),
  detail       TEXT,
  ip           VARCHAR(64),
  created_at   DATETIME DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_audit_user ON audit_log(user_id);
CREATE INDEX idx_audit_created ON audit_log(created_at);

-- -------------------------------
-- 系统配置
-- -------------------------------
CREATE TABLE IF NOT EXISTS system_config (
  id           VARCHAR(64) PRIMARY KEY,
  config_key   VARCHAR(128) NOT NULL,
  config_value TEXT,
  description  VARCHAR(256),
  updated_at   DATETIME ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_key (config_key)
);

-- -------------------------------
-- 求助查看/点击统计（可选）
-- -------------------------------
CREATE TABLE IF NOT EXISTS request_views (
  id           BIGINT AUTO_INCREMENT PRIMARY KEY,
  request_id   VARCHAR(64) NOT NULL,
  user_id      VARCHAR(64),
  view_type    VARCHAR(16) COMMENT 'list|detail',
  created_at   DATETIME DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_request_views_request ON request_views(request_id);
```

### 4.1 现有表字段扩展（可选）

若希望把“最近位置”直接放在用户表，可对 `users` 做扩展（与 `user_locations` 二选一或并存，由业务决定）：

```sql
ALTER TABLE users
  ADD COLUMN latitude      DECIMAL(10, 7) NULL AFTER address,
  ADD COLUMN longitude     DECIMAL(10, 7) NULL AFTER latitude,
  ADD COLUMN location_updated_at DATETIME NULL AFTER longitude;
```

## 5. 后端接口设计建议

### 5.1 定位与位置


| 方法   | 路径                     | 说明                                                                             |
| ---- | ---------------------- | ------------------------------------------------------------------------------ |
| POST | /api/users/me/location | 上报当前用户位置（body: latitude, longitude, address?），写入 `user_locations` 或 users 扩展字段 |
| GET  | /api/users/me/location | 返回当前用户最近一次位置（可选）                                                               |


### 5.2 志愿者匹配与附近求助


| 方法  | 路径                                  | 说明                                                    |
| --- | ----------------------------------- | ----------------------------------------------------- |
| GET | /api/volunteer/nearby-requests      | 志愿者端“附近求助”：入参当前用户位置或从 user_locations 取，按距离排序返回待接单求助   |
| GET | /api/community/volunteers/available | 现有接口可扩展：增加请求体或 query 传入 requestId/经纬度，返回按距离排序的可用志愿者列表 |


### 5.3 地理编码（可选）


| 方法   | 路径               | 说明                  |
| ---- | ---------------- | ------------------- |
| POST | /api/geo/geocode | 地址 → 经纬度（内部调高德地理编码） |
| POST | /api/geo/reverse | 经纬度 → 地址（内部调高德逆地理）  |

## 6. 前端改动要点

- **权限**：在 `module.json5` 中声明定位权限（如 `ohos.permission.LOCATION`），运行时请求用户授权。  
- **获取位置**：调用 `@ohos.geolocation` 获取经纬度；发布求助时带入表单提交。  
- **上报位置**：登录后或进入“附近求助”前，调用 `POST /api/users/me/location` 上报，便于后端做匹配。  
- **展示**：列表项/详情页展示“距离 X.X km”（由后端在接口中返回 `distance` 字段更简单，前端仅展示）。

---

## 7. 安全与配置

- **高德 Key**：仅在后端和构建/运维配置中使用，通过环境变量或配置中心注入，不提交到仓库。  
- **限流**：对地理编码/逆地理/距离接口做调用频率限制，防止滥用。  
- **隐私**：位置仅用于匹配与展示距离，需在隐私政策中说明；可做“模糊位置”（如只到区县）策略（可选）。

## 9. 如何申请高德 Web 服务 Key

后端地理编码、逆地理、距离测量需使用 **Web 服务** 类型的 Key，按以下步骤申请。

### 步骤 1：注册 / 登录

1. 打开 **高德开放平台控制台**：[https://console.amap.com/](https://console.amap.com/)
2. 若无账号，点击 **注册**（需手机号、邮箱等），完成开发者注册后登录。

### 步骤 2：创建应用

1. 在控制台左侧进入 **【应用管理】**。
2. 点击右上角 **【创建新应用】**。
3. 填写：
  - **应用名称**：如「互助帮扶后端」
  - **应用类型**：按实际选择（如「其他」或「服务」）
4. 提交后，在「我的应用」列表中会出现该应用。

### 步骤 3：添加 Key（选 Web 服务）

1. 在 **应用管理** 中，找到刚创建的应用，点击 **【添加】** 或 **【添加 Key】**。
2. 填写 Key 信息：
  - **Key 名称**：如「979048b553c25febcd9efb0549d9526c」
  - **服务平台**：必须选择 **【Web 服务】**（用于后端 HTTP 调用地理编码、逆地理、距离等）。
3. 若出现 **白名单** 或 **IP 校验**：
  - 开发阶段可先不填或填服务器出口 IP；
  - 正式环境建议配置 **IP 白名单**（仅允许后端服务器 IP 调用），提高安全性。
4. 提交后，会得到：
  - **Key**：一长串字符串，请求高德接口时用 `key=你的Key`。
  - **安全密钥**：若页面有展示，请一并保存；部分 Web 服务接口会要求配合使用（具体见高德文档）。

### 步骤 4：在项目中使用

- **不要**把 Key 写进代码或提交到 Git。
- 在本地或服务器配置中设置，例如：
  - **方式一**：在 `help` 的 `application.yml` 中增加：
    ```yaml
    amap:
      web-service-key: 你的Key
      # 若有安全密钥（部分接口需要）：
      # security-code: 你的安全密钥
    ```
    生产环境用 `application-prod.yml` 或环境变量覆盖。
  - **方式二**：用环境变量，如 `AMAP_WEB_SERVICE_KEY=你的Key`，在 `application.yml` 里用 `${AMAP_WEB_SERVICE_KEY}` 引用。

### 参考链接

- 创建应用和 Key（Web 服务）：[https://lbs.amap.com/api/webservice/create-project-and-key](https://lbs.amap.com/api/webservice/create-project-and-key)
- 控制台：[https://console.amap.com/](https://console.amap.com/)

# 手表端「互助帮扶」独立开发计划（求助者闭环）

> 目标：在**单独一套手表端**交付物上完成「求助者发起求助」的完整闭环；**不要求**与手机端功能一一对应，但求助主路径必须可用。  
> 约束：开发/验证以 **wearable** 为准；不依赖「手机端运行」作为默认验收方式（与项目 Cursor 规则一致）。

---

## 1. 背景与现状

| 项目 | 说明 |
|------|------|
| 当前工程 | `help_system` 单 `entry` 模块，`module.json5` 同时声明 `phone` 与 `wearable`。 |
| 入口分流 | `Index.ets`：手表进 `pages/watch/WatchSeekerEmergencyPage`，手机进 `SplashPage`。 |
| 已有能力 | `WatchSeekerEmergencyPage`：登录校验、定位、逆地理、`EmergencyApi` 一键发单，成功后跳转 `MyRequestsPage`（页面仍为手机向布局，需手表化或替换）。 |
| 差距 | 「单独一套手表端」若指**独立产品形态**，需在工程结构、页面集合、默认入口与构建维度上与手机端**解耦**，避免大量 `isWatchLayout` 分支长期堆叠。 |

---

## 2. 产品范围

### 2.0 手表账号与身份（固定）

- **不提供注册**：手表仅 **已有账号登录**；开户、选角、完善资料在 **手机端**完成。  
- **不切换身份**：手表不提供求助者/志愿者切换；进入能力完全由账号 **`user.role`** 决定（单角色安装包则仅服务一种角色）。

### 2.1 必须覆盖：求助者「一次求助」闭环（MVP）

以下顺序表示用户可理解的业务链路（可与实际导航略有合并）：

1. **进入应用** → 已登录且 **`role` 为求助者** 时进入求助者首页（无角色则提示去手机端设置，不在手表选角）。
2. **身份可用** → 已登录且账号具备求助者身份（与后端约定一致）。
3. **发起求助**  
   - **紧急路径**：一键/长按提交紧急求助（沿用或迭代 `EmergencyApi` + 定位/地址）。  
   - **非紧急路径（建议至少一种）**：简化为 1～2 屏的「类型 + 简短说明 + 提交」，避免复刻手机端多 Tab、长表单。
4. **可感知结果** → 提交成功反馈；可查看**本次/最近**求助状态（列表或单条详情，信息密度适配圆屏）。
5. **异常处理** → 未登录、无网络、定位失败、接口错误时有可懂提示与可恢复操作（重试、去登录）。

### 2.2 明确不做或非优先（可写在里程碑之后）

- 志愿者端、社区管理端完整功能。
- 与手机端完全一致的「首页 Tab、个人中心、全部订单列表」等。
- 复杂富文本、大图、多页向导式表单。
- 手表上「必须在手机完善资料」的硬依赖应逐步收敛（见 §6 风险）。

## 3. 技术方案选项（择一并在文档中固化）

| 方案 | 做法 | 优点 | 注意 |
|------|------|------|------|
| **A. 同仓独立模块** | 新建 `entry_watch` 或 `wearable` feature 模块，独立 `main_pages.json`、独立 Ability 入口，仅打包手表依赖 | 边界清晰，手机与手表代码物理分离 | 需调整工程 `build-profile` / 依赖关系 |
| **B. 同模块、独立路由包** | 保留单 `entry`，所有手表页放在 `pages/watch/**`，`Index` 仅加载手表子图；手机与手表共用 `services`、`models` | 改动量小，复用 API 快 | 需纪律：禁止手表页 `import` 手机大页；通过公共组件/服务协作 |
| **C. 独立应用** | 新建独立 Application / 仓库，仅对接同一后端 | 发布与版本完全独立 | 成本高，适合长期分团队维护 |

**建议**：短期用 **B** 快速闭环；若你明确「单独一套」指安装包与工程边界，则规划 **1～2 个迭代内迁到 A**。

## 4. 手表端信息架构（建议页面清单）

以下为**逻辑页面**；实现时可合并为更少 `@Component`。

| 阶段 | 页面/能力 | 说明 |
|------|-----------|------|
| W1 | `WatchSplash` / 直达 `WatchHome` | 极短品牌或直接进入主任务 |
| W2 | `WatchLogin`（或共享仅手表样式的登录） | 仅保留短信/账号能力中与后端一致的最短路径 |
| W3 | `WatchSeekerHome` | **核心**：一键紧急 + 可选「其他求助」入口；避免手机 `HelpSeekerHomePage` 整页嵌入 |
| W4 | `WatchEmergencyFlow` | 承接现有一键逻辑，可拆：确认 → 提交 → 结果 |
| W5 | `WatchSimpleRequestFlow`（可选 MVP+） | 选类型 + 一句话说明 + 提交（对接现有非紧急下单接口，若后端仅有统一接口则合并） |
| W6 | `WatchMyRequests` | 最近 1～3 条或单条状态；点击进入极简 `WatchRequestDetail` |
| W7 | `WatchSettings`（可选） | 退出登录、关于 |

**导航原则**：手表路由表独立维护（如 `watch_main_pages.json` 或模块内 profile），避免 `main_pages.json` 无限膨胀难以审计。

## 5. 后端与数据层

- **复用**：`AuthService`、`EmergencyApi`、用户资料存储 `StorageUtil`、定位与 `GeocodeApi`（若后端部署不变）。
- **约定**：请求体、Token、`help-seeker` 角色与手机端一致，便于同一套后端与运营数据。
- **文档化**：在本文档附录列出手表端实际调用的 API 列表与错误码处理策略。

---

## 6. 风险与对策

| 风险 | 对策 |
|------|------|
| 圆屏可视区域小 | 每屏单一主操作；列表用短行；关键操作字号与对比度符合无障碍可选配置 |
| 输入弱 | 少字；多用默认、定位、语音（若后续迭代）；紧急场景优先一键 |
| 手表与手机缓存独立 | 文案明确「需在本设备登录」；避免暗示必须用手机完成全流程 |
| 当前提示「在手机完善手机号」 | MVP 可保留；中长期在手表提供**极简手机号展示/确认**（只读同步自账号） |

## 7. 里程碑（建议）

| 迭代 | 交付物 |
|------|--------|
| **M0** | 确定方案 A/B/C；冻结手表 `main_pages` 与路由；定义 MVP 验收用例 |
| **M1** | 登录 + 一键紧急求助 + 成功/失败反馈（可在现有 `WatchSeekerEmergencyPage` 上收敛 UI 与跳转） |
| **M2** | `WatchMyRequests` / 极简详情；离线/弱网提示 |
| **M3** | 非紧急简流程（可选）；设置与退出；埋点或日志策略 |
| **M4** | 若采用方案 A：模块拆分与构建配置文档 |

---

## 8. 验收标准（示例）

1. 在 **手表模拟器或真机** 上，未登录用户可被引导登录，登录后能完成至少一次紧急求助提交，并看到成功反馈。  
2. 能进入「我的求助」或等价页，看到刚提交记录或状态（与后端一致）。  
3. 关闭定位权限时仍可提交（若产品允许），且文案说明清楚。  
4. 不出现仅适用于手机的布局（如不可滚动导致按钮不可见）阻碍主路径。

## 9. 与现有代码的映射（便于落地）

| 现有 | 手表独立化时的方向 |
|------|---------------------|
| `Index.ets` | 仅保留 `replaceUrl` 到手表根页面；手机逻辑可迁出或分模块 |
| `WatchSeekerEmergencyPage.ets` | 作为 W4 核心或拆子组件；跳转目标改为手表专用「我的求助」页 |
| `LoginPage.ets` | 长期改为仅 `import` 手表专用布局，或复制为 `watch/WatchLoginPage` 减少分支 |
| `MyRequestsPage.ets` | 为手表新增轻量页或条件编译，避免强依赖手机 Tab 结构 |

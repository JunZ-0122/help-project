# 验证码开发联调 + 模拟短信通知设计文档

## 1. 概述

### 1.1 目标
- 用后端真实生成的 6 位随机验证码替换当前固定 `123456` 的测试逻辑。
- 在暂时无法对接短信服务商 API 的前提下，提供一套仅用于开发和联调的“模拟短信通知”方案。
- 后端先把验证码生命周期、幂等性、过期时间、失败次数限制和一次性消费规则设计完整，后续接入真实短信平台时尽量不改业务流程。

### 1.2 当前问题
- 后端短信登录校验仍然写死 `123456`。
- `/api/auth/send-code` 只有“调用成功”语义，没有真正生成、存储、校验验证码。
- 注册流程中的验证码校验仍未实现。
- 前端页面和手表端页面都在提示测试验证码 `123456`，且验证码长度校验仍是“4 位以上”。

### 1.3 本次范围
- 登录验证码发送与校验。
- 注册验证码发送与校验。
- 开发环境模拟短信展示。
- Redis 中验证码状态设计。
- 真实短信发送通道的接口预留。

### 1.4 非目标
- 本次不对接任何第三方短信平台。
- 本次不实现复杂风控，如图形验证码、人机校验、IP 限流。
- 本次不改动已有账号密码登录主流程。

---

## 2. 方案对比

### 2.1 方案 A：前端继续写死测试验证码
- 做法：前端直接提示用户输入固定码，后端也继续按固定码校验。
- 优点：开发成本最低。
- 缺点：后端验证码链路始终是假逻辑，后续接入真实短信时需要重写发送、存储、校验、过期和失败处理，不适合作为过渡方案。

### 2.2 方案 B：前端自己生成验证码并展示
- 做法：前端随机生成 6 位数并渲染在页面上，再把这个验证码带回后端。
- 优点：表面上看能快速联调。
- 缺点：验证码真值不在后端，无法保证幂等性、过期、一致性和防篡改；后续几乎必然返工。

### 2.3 方案 C：后端真实生成验证码，开发环境返回调试信息，由前端渲染“模拟短信”
- 做法：后端生成 6 位验证码并保存到 Redis；开发环境下接口额外返回 `debugCode` 或 `mockMessage`，前端按“应用内模拟短信”展示。
- 优点：后端链路从一开始就是真实结构，后续切换到短信平台时只替换发送通道；前端也不再依赖固定测试码。
- 缺点：需要同时调整后端接口、Redis 存储和前端展示。

### 2.4 结论
- 采用方案 C。
- 这是开发成本、后续可维护性和联调效率之间最平衡的方案。

## 3. 设计原则

### 3.1 后端是真实验证码来源
- 验证码必须由后端生成。
- 前端只能展示调试信息，不能决定验证码真值。

### 3.2 调试能力与生产能力严格隔离
- 生产环境禁止返回验证码明文。
- 只有开发环境或显式开启的调试模式才允许返回 `debugCode`。

### 3.3 验证码是一次性短期凭证
- 6 位数字。
- 固定有效期。
- 校验成功后立即删除。
- 达到失败上限后失效。

### 3.4 发送接口具备幂等特征
- 在验证码仍有效且处于冷却期内时，重复点击“发送验证码”不重新生成新码。
- 开发环境下重复请求可以继续返回同一个 `debugCode`，避免用户收到多个不同验证码。

## 4. 后端设计

### 4.1 配置项
- `app.sms.mock-mode.enabled`
  - 含义：是否启用模拟短信模式。
  - 开发环境默认 `true`，生产环境默认 `false`。
- `app.sms.mock-mode.return-code`
  - 含义：接口是否返回调试验证码。
  - 开发环境默认 `true`，生产环境固定 `false`。
- `app.sms.code.ttl-seconds`
  - 默认值：`300`。
- `app.sms.code.cooldown-seconds`
  - 默认值：`60`。
- `app.sms.code.max-failures`
  - 默认值：`5`。

### 4.2 抽象分层
- `VerificationCodeService`
  - 负责生成、发送、校验、删除验证码。
- `VerificationCodeStore`
  - 负责验证码状态的读写，首个实现为 Redis。
- `SmsGateway`
  - 负责“发送通知”的抽象。
  - 本次先实现 `MockSmsGateway`，后续再替换为真实短信服务实现。

这样可以保证后续接入真实短信时，业务入口和校验逻辑不变，只替换发送网关。

### 4.2.1 密码存储策略
- 用户密码不再使用纯 `MD5` 存储。
- 统一改为 `MD5 + 随机盐值（Salt）`。
- 每个用户生成独立随机盐值，例如 16 位或 32 位随机字符串。
- 数据库存储两部分：
  - `password`：加盐后的 MD5 结果
  - `salt`：该用户对应的随机盐值
- 推荐固定计算规则，例如：
  - `passwordHash = MD5(rawPassword + salt)`
- 注册、验证码注册、短信登录自动注册、密码登录校验必须共用同一套加盐规则。

### 4.2.2 用户表字段调整
- 用户表新增盐值字段，例如 `salt` 或 `password_salt`。
- 历史用户数据如果仍是纯 `MD5`，本次采用“兼容校验 + 登录后升级”的策略。
- 具体规则：
  - 新注册用户、验证码注册用户、短信登录自动注册用户一律按 `MD5 + 随机盐值` 存储。
  - 存量纯 `MD5` 用户在密码登录时先按旧规则兼容一次。
  - 若旧规则校验成功，则立即为该用户生成随机盐值，并将密码升级为加盐后的存储格式。

### 4.3 Redis Key 设计

#### 4.3.1 验证码主体
- Key：`sms:code:{scene}:{phone}`
- Value：JSON 字符串或 Redis Hash，至少包含：
  - `code`
  - `scene`
  - `phone`
  - `sentAt`
  - `requestId`
- TTL：`300` 秒

#### 4.3.2 发送冷却
- Key：`sms:cooldown:{scene}:{phone}`
- Value：固定占位值，例如 `1`
- TTL：`60` 秒

#### 4.3.3 失败计数
- Key：`sms:fail:{scene}:{phone}`
- Value：整数
- TTL：与验证码剩余有效期一致，或不超过验证码 TTL

### 4.4 场景定义
- `login`
- `register`

不同场景使用不同 Redis key，避免注册验证码误用于登录，或登录验证码误用于注册。

### 4.5 发送验证码流程
1. 校验手机号格式与场景值。
2. 查询 `sms:code:{scene}:{phone}`。
3. 如果当前验证码仍然有效：
   - 不重新生成新验证码。
   - 若处于冷却期，直接返回成功响应，但标记这是“复用当前验证码”的结果。
   - 若开发环境开启调试返回，则继续返回相同的 `debugCode`。
4. 如果当前没有有效验证码：
   - 生成新的 6 位随机数字验证码。
   - 写入 Redis 主体 key。
   - 写入冷却 key。
   - 删除旧的失败计数 key。
5. 调用 `SmsGateway`。
   - 当前由 `MockSmsGateway` 生成模拟短信文案。
6. 返回响应。

### 4.6 发送接口响应
当前 `send-code` 返回 `Result<Void>`，本次改为返回结构化数据，例如：

```json
{
  "code": 200,
  "message": "验证码已发送",
  "data": {
    "cooldownSeconds": 60,
    "expiresInSeconds": 300,
    "reused": false,
    "mockMessage": "模拟短信：您的验证码是 381942，5 分钟内有效",
    "debugCode": "381942"
  }
}
```

约束：
- `debugCode` 只在开发环境并且显式开启配置时返回。
- 生产环境 `data` 中不应包含任何验证码明文。
- 若本次只是复用现有验证码，则 `reused=true`。

### 4.7 校验验证码流程
登录和注册共用同一套验证码校验逻辑：
1. 按 `scene + phone` 读取验证码记录。
2. 不存在则返回“验证码不存在或已过期”。
3. 比较用户输入与已存储验证码。
4. 若不一致：
   - 失败计数 `+1`。
   - 若失败次数达到上限，删除验证码主体和失败计数，并返回“验证码已失效，请重新获取”。
   - 未达到上限则返回“验证码错误”。
5. 若一致：
   - 删除验证码主体、冷却 key、失败计数。
   - 继续执行业务逻辑。

### 4.8 登录流程调整
- 短信登录不再校验固定 `123456`。
- 改为调用验证码校验服务。
- 若用户不存在，保留当前“短信登录自动注册”的业务行为，并将用户初始登录密码统一设置为固定默认密码 `123456` 的“加盐 MD5”结果。

具体规则：
- 自动注册用户时，密码字段写入 `123456` 的加盐哈希值，而不是本次短信验证码的加密值。
- 同时为该用户生成独立随机盐值并一起落库。
- 这样可以保证验证码继续保持一次性随机凭证，同时数据库中的登录密码策略保持统一，并满足论文中的密码安全要求。

### 4.9 注册流程调整
- 注册前必须校验 `register` 场景验证码。
- 校验通过后再创建用户。
- 通过验证码完成注册的用户，数据库中的登录密码同样统一写入 `123456` 的加盐哈希值，并生成独立随机盐值。
- 不再保留“验证码校验未实现”的状态。

### 4.9.1 密码登录校验调整
- 密码登录时，不再直接对输入密码做纯 `MD5` 后比较。
- 改为先读取用户盐值，再按统一规则计算加盐哈希并与数据库 `password` 字段比对。
- 这样注册用户、验证码创建用户和后续密码登录逻辑可以完全一致。

### 4.10 异常与边界处理
- 手机号非法：直接返回参数错误。
- Redis 异常：发送与校验均失败，不做前端兜底伪造。
- 验证码过期：提示重新获取。
- 多次点击发送：返回成功并复用现有验证码，不刷新有效期。
- 同一手机号跨场景：登录与注册验证码互不影响。

## 5. 前端设计

### 5.1 总体原则
- 前端不再内置 `123456` 的提示文案。
- 前端不再自己决定验证码内容。
- 只在后端返回 `debugCode` 或 `mockMessage` 时渲染“模拟短信通知”。
- 本次默认不在“发送验证码成功”提示中展示默认登录密码 `123456`；如需在注册成功后展示，作为单独 UI 文案决策处理。

### 5.2 接口模型调整
- `AuthService.sendVerificationCode()` 从返回 `boolean` 改为返回结构化结果。
- 新模型至少包含：
  - `cooldownSeconds`
  - `expiresInSeconds`
  - `reused`
  - `mockMessage?`
  - `debugCode?`

### 5.3 手机端展示
- 发送成功后保留当前倒计时行为。
- 若响应中包含 `mockMessage`：
  - 展示一条应用内调试通知。
  - 推荐形式：验证码输入框附近的提示卡片，或发送成功后的 toast + 卡片。
- 调试文案示例：
  - `模拟短信：您的验证码是 381942，5 分钟内有效。该提示仅开发环境可见。`

不建议伪装成系统短信弹窗，避免用户误解这是生产能力。

### 5.4 手表端展示
- 手表端屏幕较小，可沿用 toast 形式。
- 若后端返回 `debugCode`，toast 文案可直接显示验证码。
- 同样需要标明这是测试/调试用途。

### 5.5 验证码输入约束
- 登录页、注册页、手表登录页统一改为“严格 6 位数字验证码”。
- 删除当前“4 位以上即可”的校验。
- 错误提示文案同步调整为“请输入 6 位验证码”。

### 5.6 状态清理
- 用户重新发送验证码时，页面上的模拟短信内容更新为最新一次有效结果。
- 登录成功、注册成功或离开页面后，清理本地调试提示状态。

## 6. 测试设计

### 6.1 后端单元测试
- 首次发送生成 6 位验证码并写入 Redis。
- 冷却期内重复发送不生成新验证码，返回 `reused=true`。
- 验证码校验成功后删除 Redis 状态。
- 验证码校验失败时失败计数递增。
- 达到最大失败次数后验证码失效。
- 关闭 mock 返回时响应中不包含 `debugCode`。

### 6.2 联调验证
- 登录页发送验证码，页面展示模拟短信。
- 注册页发送验证码，页面展示模拟短信。
- 输入正确验证码可登录或注册。
- 输入错误验证码超过上限后必须重新发送。
- 60 秒内重复点击发送，不应出现新验证码。

### 6.3 回归验证
- 账号密码登录不受影响。
- 自动注册用户的角色选择流程不受影响。
- 通过验证码创建的用户，后续可使用密码 `123456` 进行密码登录。
- 密码登录校验应基于用户各自的随机盐值完成，而不是纯 `MD5` 比较。
- 手表端短信登录流程可继续使用。

## 7. 风险与防护

### 7.1 风险：调试验证码泄露到生产环境
- 防护：生产环境默认关闭 `mock-mode` 和 `return-code`。
- 防护：增加测试断言，确保生产配置下响应不含 `debugCode`。

### 7.2 风险：重复发送导致用户拿到多个验证码
- 防护：有效期内复用现有验证码，不重复生成。

### 7.3 风险：验证码被长期复用
- 防护：固定 TTL、成功后删除、失败超限后删除。

### 7.4 风险：验证码创建用户使用统一默认密码 `123456`
- 防护：默认密码只写入数据库，不参与短信验证码返回或页面调试展示。
- 防护：数据库不存明文，也不存纯 `MD5`，统一使用 `MD5 + 随机盐值`。
- 防护：验证码仍保持随机、短期、一次性消费，不与登录密码复用。
- 防护：后续可再补充“首次登录后改密”或“个人中心修改密码”能力，但不属于本次范围。

## 8. 实施顺序建议
1. 后端新增验证码服务、Redis 存储和 mock 发送网关。
2. 改造 `send-code` 接口返回结构化结果。
3. 改造登录与注册验证码校验逻辑。
4. 改造前端 `AuthService` 返回模型。
5. 更新手机端与手表端提示 UI。
6. 补充测试与联调验证。

---

## 9. 结论
- 当前最合理的过渡方案不是“继续使用固定验证码”，也不是“前端自己伪造验证码”，而是“后端真实生成并保存验证码，前端仅渲染开发环境模拟短信”。
- 该方案能满足当前没有短信 API 的现实约束，同时保住后续接入真实短信服务时最关键的后端结构与安全边界。

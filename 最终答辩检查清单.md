# 最终答辩检查清单（高德定位与匹配）

> 目标：10-15 分钟内完成“可演示、可复现、可解释”的最终检查。

## A. 必做（阻塞项）

- [ ] **高德 Key 已切换为 Web服务**
  - 控制台 `help` 应用新增 Key，`绑定服务=Web服务`
  - `help/src/main/resources/application.yml` 中 `amap.web-service-key` 已替换
- [ ] **后端已重启**
  - 看到 Spring Boot 启动成功日志，端口 `8080`
- [ ] **逆地理接口可用**
  - 打开 `http://localhost:8080/demo/location.html
  - 输入经纬度并点击解析，返回有效地址（非空）
  - 后端日志不再出现 `USERKEY_PLAT_NOMATCH`

## B. 业务链路验收（核心）

- [ ] **位置上报**
  - 登录后调用 `POST /api/users/me/location` 成功
  - `GET /api/users/me/location` 可查到经纬度
- [ ] **求助创建（带经纬度）**
  - 前端授权定位后发布求助，请求体含 `latitude/longitude`
- [ ] **求助创建（仅地址）**
  - 不传经纬度仅传地址，后端 geocode 补全成功
- [ ] **附近求助按距离**
  - 志愿者端调用 `GET /api/volunteer/nearby-requests`
  - 返回结果按距离排序并展示 `distance`
- [ ] **社区可用志愿者按距离**
  - `GET /api/community/volunteers/available?requestId=...`
  - 列表按距离排序并展示距离

## C. 演示准备（给老师看）

- [ ] **静态演示页准备好**
  - 地址：`http://localhost:8080/demo/location.html`
  - 准备两组经纬度（例如北京、深圳）现场切换
- [ ] **前端页面可展示**
  - 紧急求助页“获取位置”可触发定位/逆地理
  - 模拟器场景可说明“GPS模拟设置后再获取位置”
- [ ] **关键日志可见**
  - `[Geocode] regeo 请求/结果`
  - `[User] 上报位置`
  - `[Volunteer] 附近求助结果`

## D. 数据库抽查（可信证明）

- [ ] `user_locations` 有当前用户最近坐标
- [ ] `help_requests` 新创建记录的 `latitude/longitude` 非空（至少一条）

## E. 讲解话术（30 秒）

- “前端拿到经纬度后调用后端，后端使用高德 Web 服务做逆地理/地理编码。”
- “定位结果会入库，附近求助与志愿者分配都按距离排序。”
- “静态页可独立证明后端高德链路已通；业务页可证明端到端链路已通。”

---

## 常见失败与处理

- `USERKEY_PLAT_NOMATCH`：用成了 `Web端` Key，改成 `Web服务` Key 并重启后端。
- 模拟器“获取不到位置”：确认是“模拟器运行”（不是 Preview），并先在 GPS 模拟里设置坐标并确定。
- 有经纬度但地址为空：先用演示页验证 `/api/geocode/regeo`，再检查后端网络与 Key 配置。

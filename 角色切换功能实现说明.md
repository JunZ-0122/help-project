# 角色切换功能实现说明

## 功能概述

为了满足用户在不换账号的情况下切换角色的需求，系统新增了角色切换功能。用户可以在个人中心切换到其他角色，同时保留原角色下的所有历史数据。

## 设计原则

1. **数据保留**：切换角色后，保留用户在原角色下的所有历史数据
2. **权限变更**：切换后立即生效新角色的权限
3. **二次确认**：切换前需要用户确认，并提示可能的影响
4. **审计追踪**：记录角色切换历史（可扩展）

## 实现内容

### 1. 后端接口

#### 新增接口：切换角色

**接口地址**: `PUT /api/users/me/role/switch`

**请求头**:
```
Authorization: Bearer {token}
Content-Type: application/json
```

**请求体**:
```json
{
  "role": "community"  // help-seeker, volunteer, community
}
```

**响应示例**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "id": "user004",
    "name": "赵六",
    "phone": "13800138004",
    "role": "community",  // 已更新
    "status": "online",
    "updatedAt": "2026-03-08T00:15:30"
  }
}
```

**错误响应**:
```json
{
  "code": 500,
  "message": "当前已是该角色，无需切换"
}
```

#### 保留接口：首次选择角色

**接口地址**: `PUT /api/users/me/role`

此接口保持不变，仅用于首次选择角色（注册后未选择角色的用户）。

#### 新增 API 方法

**文件**: `help_system/entry/src/main/ets/services/ApiService.ets`

```typescript
// 切换用户角色（允许已有角色的用户切换）
static async switchUserRole(role: string): Promise<UserProfile> {
  const requestData: UpdateRoleData = { role: role };
  const response = await HttpClient.put<UserProfile>('/users/me/role/switch', requestData);
  
  if (response.data) {
    return response.data;
  }
  throw new Error('切换角色失败：响应中没有数据');
}
```

## 使用流程

### 用户操作流程

1. **进入个人中心**
   - 用户在当前角色的主页点击"个人中心"或"我的"

2. **选择切换角色**
   - 在个人中心页面点击"切换角色"菜单项
   - 进入角色切换页面

3. **查看可选角色**
   - 页面显示当前角色和所有可选角色
   - 每个角色都有图标、名称、描述

4. **选择新角色**
   - 点击目标角色的"切换"按钮
   - 弹出确认对话框

5. **确认切换**
   - 对话框显示切换后的影响提示
   - 点击"确认切换"按钮

6. **自动跳转**
   - 切换成功后显示成功提示
   - 自动跳转到新角色的主页

### 技术流程

```
用户点击切换
    ↓
前端显示确认对话框
    ↓
用户确认
    ↓
调用 PUT /api/users/me/role/switch
    ↓
后端验证角色有效性
    ↓
检查是否切换到相同角色
    ↓
更新数据库 users.role
    ↓
返回更新后的用户信息
    ↓
前端更新本地存储
    ↓
跳转到新角色主页
```

## 数据影响说明

### 求助者 → 志愿者

**保留数据**:
- 历史求助记录（help_requests 表）
- 聊天消息（chat_messages 表）
- 评价记录（reviews 表）

**新增权限**:
- 可以接单服务他人
- 可以查看志愿者订单
- 可以获得服务评分

**失去权限**:
- 无法发布新的求助请求（但可以查看历史）

### 志愿者 → 求助者

**保留数据**:
- 历史服务记录（volunteer_orders 表）
- 服务评分和认证资质
- 志愿服务时长统计

**新增权限**:
- 可以发布求助请求
- 可以与志愿者沟通

**失去权限**:
- 无法接单服务他人（但保留历史服务记录）

### 任意角色 → 社区管理员

**保留数据**:
- 原角色的所有历史数据

**新增权限**:
- 管理所有求助请求
- 分配志愿者
- 查看数据统计

**失去权限**:
- 根据原角色而定

## 注意事项

### 1. 数据一致性

- 角色切换不会删除任何历史数据
- 用户在不同角色下的数据相互独立
- 切换回原角色后，原数据仍然可见

### 2. 权限控制

- 切换角色后，Token 中的 role 字段不会立即更新
- 建议切换后重新登录，或实现 Token 刷新机制
- 后端接口应始终以数据库中的 role 为准

### 3. 业务规则

- 不允许切换到当前相同的角色
- 切换前必须二次确认
- 切换后自动跳转到新角色主页

### 4. 扩展建议

**角色切换日志**（可选）:
```sql
CREATE TABLE role_change_logs (
  id VARCHAR(32) PRIMARY KEY,
  user_id VARCHAR(32) NOT NULL,
  old_role VARCHAR(20) NOT NULL,
  new_role VARCHAR(20) NOT NULL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_user_id (user_id)
);
```

**切换次数限制**（可选）:
- 限制每天/每月的切换次数
- 防止频繁切换影响系统稳定性

**切换冷却期**（可选）:
- 切换后 24 小时内不允许再次切换
- 给用户适应新角色的时间
